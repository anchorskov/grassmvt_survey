Security, Privacy, and Trust

We built GrassrootsMVT to gather civic input without letting spam, bots, or coordinated manipulation drown out real voices. We also protect privacy by collecting only what we need to run the platform and publish results with integrity.

This page explains:

  • how accounts and sessions work
  • how surveys are published and versioned
  • how responses are stored and counted
  • how location and address verification works, and what we store and do not store
  • how results are displayed by tier
  • how we audit the system

Our reporting tiers

We publish results in tiers so the public can see the difference between broad participation and higher confidence participation.

Tier 1: General Public, no verified address

This tier includes responses from all accounts including those that have not completed address verification.

Tier 2: General Public, verified address

This tier includes responses from accounts that completed address verification.

From this tier, we offer filters so we can see results at different levels:

  • State
  • Federal House District
  • State House District
  • State Senate District

We display these filters as a dropdown so we can view the same verified address dataset at different jurisdiction levels. Our plan is to scale down to wards within cities and counties as those data become available.

What we store

Account basics

  • Email
  • Password hash, not the password
  • Session records to keep us logged in across page loads

Session timeouts

We use two timeouts to keep accounts secure.

  • Idle timeout ends a session after a period of inactivity.
  • Absolute timeout ends a session after a fixed number of days.

Survey integrity

  • The exact survey definition used, SurveyJS JSON
  • Survey versions and hashes so wording changes are visible and auditable
  • Response records with timestamps

Response editing without duplicate counting

  • Each account has one counted response per survey version.
  • Users can revise answers.
  • Revisions update the existing response and create an audit record.
  • The survey list shows completion date and edit status.

Jurisdiction keys for aggregation

When we verify an address, we store the jurisdiction keys needed for aggregation, for example:

  • state
  • federal house district
  • state house district
  • state senate district

We store these keys so responses can aggregate correctly across multiple sessions.

Audit trail

We record security relevant events so we can investigate problems and show a clear history of what happened.

What we do not store

Passwords

We never store plaintext passwords.

Precise device location

If we run a home of record proximity check, we do not store your location. We store only outcomes and buckets, for example:

  • pass or fail
  • distance bucket
  • accuracy bucket
  • timestamp

Verification, plain language

Verification helps us reduce manipulation without turning the platform into a surveillance system.

Address verification outcome

For verified address tiering, we focus on three things:

  1. The address we use to derive districts
  2. The districts we derive from that address
  3. A verification outcome, stored as status plus audit events

Wyoming

Wyoming provides candidates access to a voter file. For Wyoming, we can verify eligibility using that dataset and then store a salted hash that prevents one person from verifying multiple accounts. Candidates wishing to participate may contact me Jimmy Skovgard via email at skovgard2026.org.

We do not publish or track personally identifying information. We use the voter file data provided by your elected secretary of state.

National

For broader participation across the country, we use an address based jurisdiction method. We ask for an address, derive districts, and store only the jurisdiction keys needed for aggregation. We can also store an address hash for deduplication signals and for internal consistency checks.

If we add a proximity check, we ask the user to verify location once at the home of record address. We do not store your location, latitude, or longitude.

Anti spam protections

We use multiple layers:

  • bot challenge on sensitive actions, signup and other high risk endpoints
  • rate limits on abuse prone endpoints
  • one counted response per user per survey version, unless the survey design explicitly allows updates
  • audit events that flag suspicious patterns

Survey transparency and versioning

We treat survey wording as a public artifact.

  • We publish surveys from JSONC source file for use on grassrootsmvt.org.
  • Each version gets a hash.
  • Any change to wording creates a new version and a new hash.

This means anyone can compare versions and see what changed.

Your control and consent

  • Participation stays optional.
  • Verification stays optional.
  • Verified tiers strengthen aggregation quality.
  • We keep location checks one time and store only outcomes.
  • You will receive an email receipt with a token which you may use to revise your answers unless the survey sunsets. Most surveys are dynamic, not all.

Feedback

Security is not a static promise, it is a process. We invite feedback and we publish the results.
